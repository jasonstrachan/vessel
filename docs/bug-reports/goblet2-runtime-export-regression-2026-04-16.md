# Goblet2 Runtime Export Regression

Date: 2026-04-16

## Summary

Goblet export regressions were bisected to the feature set introduced between `18f8d5fab` and `926b540f0`, but selective restoration isolated the break to:

- `public/goblet2/goblet2.js`
- `public/goblet2/goblet2-inline.js`

The display-filter and film-noise pipeline changes were restored independently and did not reintroduce the bug.

For the current single-file Goblet2 export regression, the confirmed root cause is in the inline runtime builder:

- `scripts/build-goblet-runtime.mjs`

The builder was flattening `src/lib/displayFilterPipeline.js` into the same module scope as `public/goblet2/goblet2.js`. Both files defined helper names such as `clamp01`, which caused the generated single-file runtime to fail at parse time with:

- `Uncaught SyntaxError: Identifier 'clamp01' has already been declared`

That failure happened before Goblet2 could render anything, which is why the exported canvas appeared blank.

## Provenance

`public/goblet2/goblet2-inline.js` is not a hand-maintained source file. It is generated by:

- `npm run build:goblet-inline`
- `scripts/build-goblet-runtime.mjs`

For Goblet2, that generator inlines:

- `public/goblet2/goblet2.js`
- `public/goblet2/alignFitResolver.js`
- `public/goblet2/num.js`
- `public/goblet2/fflate-inflate.js`
- `src/lib/displayFilterPipeline.js`

The export pipeline then embeds `public/goblet2/goblet2-inline.js` into single-file Goblet2 exports via `src/utils/export/webglExporter.ts`.

This matters because the visible failure artifact was `goblet2-inline.js`, but the source of the regression was upstream. Diffing the generator inputs between `926b540f0` and `3e954dc2f` showed:

- `public/goblet2/goblet2.js` changed
- `src/lib/displayFilterPipeline.js` did not change
- `scripts/build-goblet-runtime.mjs` did not change

Conclusion: the bad `goblet2-inline.js` came from regenerating against a changed `public/goblet2/goblet2.js`, not from a change in the inline builder or the shared display-filter pipeline.

That conclusion applied to the earlier Goblet2 runtime bisect. The later single-file blank-canvas regression investigated on 2026-04-16 was different: it came from the inline builder concatenating two modules into one scope without isolating private helpers.

## Symptoms

- Exported Goblet output appeared blank or missing expected layer content.
- Restoring only the Goblet2 runtime files from `HEAD` was sufficient to make exports fail again.
- Keeping the Goblet2 runtime files on the `18f8d5fab` version restored working exports.

## Isolated Runtime Differences

The offending runtime diff included two behavior changes in Goblet2:

1. Removing the viewer-only CC time multiplier in `BrushWebGLRenderer.render(...)`.
2. Changing `ColorCycleLayerPlayer.advance(...)` to use target-FPS bucketed accumulation instead of direct `deltaSeconds` stepping.

These changes are guarded by regression tests in:

- `tests/goblet2-runtime-regression.test.ts`

## Current Safe State

- Keep `public/goblet2/goblet2.js` and `public/goblet2/goblet2-inline.js` pinned to the `18f8d5fab` behavior.
- The display-filter / film-noise files can remain on current `HEAD`.
- Keep `scripts/build-goblet-runtime.mjs` wrapping the display-filter pipeline in its own IIFE export block before minification.

## Regression Tests

The fix is now guarded in two places:

- `tests/goblet2-runtime-regression.test.ts`
  - keeps the known-safe Goblet2 timing behavior
  - asserts the inline runtime scopes the display-filter pipeline in an IIFE
  - asserts the top-level `clamp01` collision no longer exists in `public/goblet2/goblet2-inline.js`
- `tests/goblet-display-filters-runtime.test.ts`
  - asserts both Goblet1 and Goblet2 inline runtimes scope the display-filter pipeline the same way
- `tests/goblet2-single-file-smoke.spec.ts`
  - launches a real browser page with a single-file Goblet2 document built from `public/goblet2/goblet2-inline.js`
  - fails on page errors, console errors, or a zero-pixel canvas
  - specifically guards the parse-time blank-canvas class of regression
  - runnable via `npm run test:goblet2:single-file-smoke`

## Why The Generated File Looked Like The Bug

- Single-file Goblet2 exports execute `public/goblet2/goblet2-inline.js`, so that file is the runtime artifact users actually see fail.
- Because it is generated, it can appear to be the only changed Goblet file even when the real cause is a source-runtime change upstream.
- In this incident, `goblet2-inline.js` accurately reflected the state of `public/goblet2/goblet2.js` at generation time.

For the parse-time blank-canvas regression, `goblet2-inline.js` was also the visible failure artifact, but the underlying source bug was in how `scripts/build-goblet-runtime.mjs` composed the final module.

## Follow-up

If Goblet2 runtime improvements need to be reintroduced, do so by replaying the `goblet2.js` / `goblet2-inline.js` diff in small hunks and testing export after each step.
