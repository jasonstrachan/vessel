# Plan: Canonical Dither Pipeline for Sequential Layers

Date: 2026-02-10  
Status: Proposed

## Problem Statement

Sequential layer dither output is currently generated by a separate replay/materializer path, so it cannot match normal layer dither output exactly.

Current code confirms this split:
- Normal/layer draw path uses region post-process dithering in `src/hooks/brushEngine/strokeDitherRegion.ts`.
- Sequential replay path synthesizes dither in `src/lib/sequential/materializer/SequentialCpuMaterializer.ts`.
- "GPU" sequential materializer currently delegates to CPU logic in `src/lib/sequential/materializer/SequentialGpuMaterializer.ts`.

## Root Cause

Two different algorithms are producing "dither":
- Canonical draw-time dither: `ditherRegionWithCurrentPressure(...)` with `applyDithering*` and bg-fill/no-bg handling.
- Replay-time synthetic dither: `shouldKeepTexturedPixel(...)` + per-pixel acceptance in `paintStamp(...)`.

Because these are not the same kernel and do not operate on the same intermediate pixels, parity is structurally impossible.

## Target Architecture

Single-source semantics:
1. One canonical dither kernel module used by both normal and sequential paths.
2. Sequential capture records canonical raster patch artifacts for dither strokes (not abstract stamp replay only).
3. Sequential playback/export consumes recorded patch artifacts directly for dither events.
4. GPU remains acceleration/presentation only; it never changes semantics.
5. No silent fallback to synthetic sequential dither for dither brushes. Fail fast in development, explicit guard in production path.

## Data Model Changes

Add optional canonical raster patch payload to sequential events (for dither-capable strokes):
- `SequentialRasterPatchV1`
  - `version: 1`
  - `bounds: { x, y, width, height }`
  - `pixelFormat: 'rgba8'`
  - `premultipliedAlpha: true`
  - `colorSpace: 'srgb'`
  - `rgbaBase64: string` (or chunked tile payload in later optimization pass)

Update:
- `src/types/index.ts` (`SequentialStrokeEvent`)
- `src/lib/sequential/SequentialStrokeChunk.ts` encode/decode
- payload budgeting logic in `src/lib/sequential/SequentialPayloadBudget.ts`

## Execution Plan

### Phase 1: Canonical Kernel Extraction (No Behavior Change)
- Extract reusable dither kernel API from `strokeDitherRegion` into shared module, e.g.:
  - `src/lib/dither/canonicalStrokeDitherKernel.ts`
- Keep existing normal-layer behavior unchanged by adapting `strokeDitherRegion` to call the extracted kernel.
- Add unit tests against current expected outputs to lock behavior.

Exit criteria:
- Normal drawing renders identical output before/after extraction.
- `npm run type-check`, `npm run lint`, `npm test` pass.

### Phase 2: Sequential Capture of Canonical Patches
- In capture path (`src/hooks/canvas/handlers/strokeBatching.ts` + `src/hooks/canvas/handlers/sequential/sequentialCapture.ts`):
  - For dither events, generate canonical patch from the same kernel and attach patch payload to event.
  - Keep stamp payload for non-dither brushes unchanged.
- Ensure event buffering/flush supports mixed events (patch + stamp-only).

Exit criteria:
- Sequential event log persists canonical patches for dither brushes.
- Hard/soft payload limits account for patch bytes correctly.

### Phase 3: Materializer Consumption Switch
- In `src/lib/sequential/materializer/SequentialCpuMaterializer.ts`:
  - For dither events with canonical patch payload, composite patch directly.
  - Remove/disable replay dither synthesis for those events.
- Add guard path:
  - If event requires canonical dither but patch missing, throw in dev and mark fallback telemetry in prod (no synthetic branch).

Exit criteria:
- Dither frame output comes from patch payload path.
- No path remains that "recreates" dither from stamp coverage for dither brushes.

### Phase 4: Renderer/Backend Policy and Export Parity
- Keep backend contract stable in `src/lib/sequential/SequentialLayerRenderer.ts`.
- Keep `SequentialGpuMaterializer` semantics CPU-equivalent until true GPU parity suite exists.
- Ensure export path consumes same rendered frame cache/patch output, not alternative dither logic.

Exit criteria:
- Vessel viewport and export outputs match for dither sequential frames.

### Phase 5: Delete Legacy Sequential Dither Synthesis
- Remove obsolete dither replay code from materializer (`shouldKeepTexturedPixel` dither branch, replay palette synthesis path for dither events).
- Keep non-dither texture/plugin paths intact.

Exit criteria:
- Codebase has one dither semantics path for drawing and sequential playback.

## Verification Strategy

Add parity tests that compare pixel output between:
- Normal layer canonical dither render.
- Sequential frame materialization render using canonical patch events.

Matrix:
- Algorithms: `sierra-lite`, `atkinson`, `bayer`, `blue-noise`, `void-and-cluster`
- `ditherBackgroundFill`: on/off
- `fillResolution`: 1, 2, 4
- Tip shapes: square, round, triangle/diamond mapping
- Blend modes: `source-over`, `destination-out`

Assertions:
- Pixel-identical in deterministic scenarios.
- Bounding box/alpha coverage equality where deterministic equivalence is expected.

## Risk Controls

- Introduce feature flag for phased rollout (default on in dev, gated in prod initially).
- Keep migration compatible: old events without patch continue for non-dither only.
- Budget guardrails:
  - monitor payload growth and cache churn after patch adoption.
  - enforce existing 32 MB warning and 96 MB cap with patch-size-aware accounting.

## GPU vs CPU Policy (Explicit)

- CPU canonical semantics define truth.
- GPU may optimize transport/compositing only.
- `SequentialGpuMaterializer` must not introduce alternative dither math.
- GPU enablement for canonical dither path requires parity tests to pass first.
